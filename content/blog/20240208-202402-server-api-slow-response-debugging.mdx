## 로딩 스피너가 1초 이상 돈다

코인니스 서비스의 라이브 페이지에는 인플루언서를 구독하면 그 피드를 리스트로 보여주는 기능이 있다. 어느 날 개발 중에 리스트를 조회하는데 로딩 스피너가 1초 이상 뜨는 현상을 발견했다.

## 원인 추적

아무리 생각해도 클라이언트에서 화면을 의도적으로 느리게 그리지 않는 이상, 로딩 스피너가 1초 이상 노출될 일은 없다. 그렇다면 서버에서 데이터를 응답하기까지의 시간이 오래 걸린다고 판단했다.

개발자 도구 Network 탭에서 API에 마우스를 올리면 **Waiting for server response** 막대그래프가 나온다. 이 부분이 서버 응답 대기 시간이다.

가설이 적중했다. 내 네트워크 환경 때문일 수도 있으니까, 옆에 앉은 프론트 팀원에게 확인을 부탁했다. 근데 팀원 쪽에서는 로딩 스피너가 보이지도 않고 피드가 바로 불러와졌다. 서버 응답도 0.2초 만에 왔다.

## 특정 인플루언서만 느리다

나만 느린 건가? 구독한 인플루언서가 많아서 그런가? 아니었다. 인플루언서 한 명만 구독해도 1초 이상 걸렸다.

구독 추가, 취소, 조회를 다양한 케이스로 시도해본 결과, **특정 인플루언서의 피드만 느렸다.**

![서버-api-call이-느린데-말해야하나-02.png](/blog/images/20240208-202402-server-api-slow-response-debugging-01.png)

`Ryan Selkis`를 구독하고 피드를 조회하면 첫 페이지를 불러오는 데 **0.15초**. 정상이다.

![서버-api-call이-느린데-말해야하나-03.png](/blog/images/20240208-202402-server-api-slow-response-debugging-02.png)

`BSC News`를 구독하고 피드를 조회하면 첫 페이지를 불러오는 데 **1.18초**. 스웨거에서도 똑같은 속도로 측정됐다. 클라이언트 문제가 아니라 서버 문제가 맞다.

## 말해야 하나?

원인 파악은 했고, 서버 쪽에서 성능 개선이 필요한 문제라는 결론이 났다. 근데 고민이 됐다. 내가 말 안 해도 백엔드 개발자들이 이미 알고 있지 않을까? 2년 동안 개발하면서 서버 응답이 느렸던 적이 없어서, 프론트 팀원들도 이걸 공유해야 하는 건지 확답을 못 하는 상황이었다.

## 돌이켜보면

이번에 내가 잘했다고 생각하는 건 세 가지다.

1. 데이터를 조회하는 화면이 느리다는 걸 **문제로 인식**한 점. 문제로 인식하지 않았다면 유저들도 데이터를 보는 데 1초 이상 기다려야 했을 수 있다.
2. 클라이언트가 아니라 **서버 이슈**라는 결론을 내린 것.
3. "API 느려요! 고쳐주세요!"가 아니라, 다양한 케이스를 시도해보고 **어떤 상황에서 느린지까지 정리해서 공유**한 것. 백엔드 개발자 분들이 디버깅하는 시간을 줄일 수 있었을 거다.
