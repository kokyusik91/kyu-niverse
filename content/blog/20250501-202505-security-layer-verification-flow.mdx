## 인증 조합이 단순하지 않았다

코인니스에 거래소 서비스를 새로 붙이면서, 금융 서비스 수준의 보안 검증이 필요해졌다. 출금, 출금 주소 등록, 비밀번호 재설정, 회원 탈퇴 같은 민감한 동작마다 다단계 인증을 거쳐야 한다.

여기까지는 별로 안 복잡해 보인다. 근데 실제로 따져보면 이렇다.

- **인증 수단이 3종류**: 자산 비밀번호, Google OTP, 이메일 인증
- **유저 상태에 따라 흐름이 갈린다**: OTP를 등록한 유저는 자산비밀번호 → OTP, 안 한 유저는 자산비밀번호 → 이메일
- **자산 비밀번호 미설정 유저**: 검증 플로우에 들어가기 전에 설정부터 시켜야 한다
- **5회 실패 시 Lock**: 재설정 플로우로 전환
- **기능마다 검증 후 동작이 다르다**: 출금은 서버에서 `requestId`를 받아 다음 단계로 넘겨야 하고, 출금 주소 등록은 검증 성공 후 바로 API 호출

이걸 기능마다 개별 구현하면 인증 로직이 곳곳에 흩어진다. 새 기능 추가할 때마다 같은 검증 코드를 복사하게 될 게 뻔했다. 추상화가 필요했다.

## 어떤 레이어에서 추상화할 것인가

일단 정리가 필요했다. 이 인증 플로우에서 **변하는 것**과 **변하지 않는 것**을 나눠봤다.

**변하지 않는 것:**
- 자산 비밀번호 → OTP 또는 이메일, 이 순서 자체
- 각 인증 단계의 UI (OTP 6자리 입력, 이메일 인증 코드 입력)
- 에러 처리 패턴 (400: 틀림 + 시도 횟수, 423: Lock)

**변하는 것:**
- 검증 성공 후 실행할 동작 (출금 API vs 주소 등록 API vs ...)
- Step 간 넘기는 데이터 (출금은 `WithdrawResponse`를 들고 다녀야 하고, 주소 등록은 없음)
- 자산 비밀번호 미설정 시 분기

이걸 보고 3개 레이어로 나눴다.

1. **AssetPasswordRequired** — 자산 비밀번호 미설정 유저를 걸러내는 가드
2. **VerifyDialog** — 검증 모달의 공통 껍데기
3. **Funnel** — 실제 인증 단계 흐름

## UI 컴포넌트부터 만들었다

처음부터 전체 구조를 설계하지 않았다. 먼저 만든 건 개별 인증 UI 컴포넌트와 공통 훅이었다.

```tsx
// OTPInputUI — 순수 UI 컴포넌트
export function OTPInputUI({ value, onChange, onComplete, title, errorMessage, resetErrorMessage }: OTPInputUIProps) {
  return (
    <div>
      {title && <h1 className="text-title-02-20-b text-text-txt-06 py-3">{title}</h1>}
      <p className="text-body-02-15-r text-text-txt-04 mb-1">구글 OTP</p>
      <p className="text-body-02-15-r text-text-txt-03 mb-6">숫자 6자리를 입력해 주세요.</p>
      <InputOTP autoFocus maxLength={6} value={value}
        onChange={(string) => { onChange(string); resetErrorMessage(); }}
        onComplete={onComplete}>
        {/* slots */}
      </InputOTP>
      {errorMessage && <span className="text-keycolor-system-red">{errorMessage}</span>}
    </div>
  );
}
```

`OTPInputUI`는 상태 관리를 안 한다. 값과 핸들러를 전부 밖에서 받는다. 같은 방식으로 `AssetPasswordInput`도 만들었다. 이렇게 해둬야 어떤 Funnel에서든 같은 UI를 가져다 쓸 수 있다.

검증 로직은 커스텀 훅으로 뺐다.

```ts
// useAssetPasswordVerification
export default function useAssetPasswordVerification({
  onSuccessAfterVerify,
  onErrorAfterVerify,
}: useAssetPasswordVerificationProps) {
  const [value, setValue] = useState('');
  const [trialCount, setTrialCount] = useState<FailCountDto | undefined>(undefined);
  const [errorMessage, setErrorMessage] = useState<string | undefined>(undefined);
  const [isLocked, setIsLocked] = useState(false);

  const { mutate: verifyCode } = useMutation({
    mutationFn: verifyAssetPassword,
    onSuccess: async () => {
      await onSuccessAfterVerify(value);
    },
    onError: (error: unknown) => {
      if (isAxiosError<VerifyAssetPasswordError<FailCountDto>>(error)) {
        const statusCode = error.response?.status;
        // 400: 틀림 + 시도 횟수
        if (statusCode === BAD_REQUEST_ERROR_CODE) {
          if (data?.code === TRY_COUNT_CODE && data?.data) {
            setTrialCount({ ...data.data });
          }
          setErrorMessage(message || '');
          return;
        }
        // 423: Lock
        if (statusCode === LOCKED_ERROR_CODE) {
          setIsLocked(true);
          setErrorMessage(message || '');
        }
      }
    },
  });

  return { value, setValue, handleComplete, trialCount, errorMessage, isLocked };
}
```

400(시도 횟수 포함), 423(Lock) 같은 에러 분기를 훅 하나에 모아뒀다. 이 훅을 쓰는 컴포넌트는 에러 코드를 알 필요가 없다.

## Funnel 패턴으로 다단계 흐름 관리

인증 플로우는 전형적인 다단계 순차 흐름이다. 이런 흐름을 관리하는 데 Funnel 패턴이 잘 맞는다. Funnel 패턴이란 여러 Step을 순서대로 밟아가는 UI를 하나의 상태 머신처럼 관리하는 방식이다. Toss의 `useFunnel`을 사용했다.

```ts
const funnel = useFunnel<{
  [FUNNEL_STEPS.assetPassword]: AssetInputType;
  [FUNNEL_STEPS.otp]: OTPInputType;
  [FUNNEL_STEPS.email]: EmailInputType;
}>({
  id: 'asset-password',
  initial: {
    step: FUNNEL_STEPS.assetPassword,
    context: { hasOtp },
  },
});
```

각 Step의 context 타입이 명시되어 있어서, `otp-step`에서는 `data: WithdrawResponse`가 반드시 존재한다는 걸 타입 레벨에서 보장할 수 있다. Step 간 데이터 전달이 타입 안전하게 이루어진다는 뜻이다.

## 핵심 — render-props로 "검증"과 "동작"을 분리

여기가 가장 고민한 부분이다. 출금과 출금 주소 등록은 검증 단계가 같다. 자산비밀번호 → OTP 또는 이메일. 근데 검증 후 실행할 동작이 다르다.

- **출금**: 자산비밀번호 검증 → 서버에서 `WithdrawResponse`(requestId 포함) 수신 → 이 데이터를 OTP/이메일 단계로 전달 → 최종 검증
- **출금 주소 등록**: 자산비밀번호 검증 → OTP/이메일 검증 → 바로 주소 등록 API 호출

Funnel 컴포넌트가 검증 흐름만 관리하고, 각 Step의 구체적인 렌더링은 외부에서 주입하도록 render-props 패턴을 썼다. render-props란 컴포넌트의 children이나 props로 함수를 넘겨서, 렌더링할 내용을 외부에서 결정하게 하는 패턴이다.

```tsx
interface WithdrawalFunnelProps {
  closeModal: () => void;
  setCurrentModalMetaInfo: (value: ModalMetaInfo) => void;
  renderSteps: {
    assetPassword: ({ onNext, closeModal }: ...) => JSX.Element;
    otp: ({ withdrawRequestData, closeModal }: ...) => JSX.Element;
    email: ({ withdrawRequestData, closeModal }: ...) => JSX.Element;
  };
}
```

Funnel 내부에서는 Step 전환과 데이터 전달만 담당한다.

```tsx
switch (funnel.step) {
  case FUNNEL_STEPS.assetPassword:
    return renderSteps.assetPassword({ onNext, closeModal });
  case FUNNEL_STEPS.otp:
    return renderSteps.otp({ withdrawRequestData: funnel.context.data, closeModal });
  case FUNNEL_STEPS.email:
    return renderSteps.email({ withdrawRequestData: funnel.context.data, closeModal });
}
```

사용하는 쪽에서는 이렇게 된다.

```tsx
<AssetPasswordRequired open={open} setOpen={setOpen}>
  {({ open, setOpen }) => (
    <VerifyDialog open={open} onOpenChange={setOpen}>
      {({ closeModal, setCurrentModalMetaInfo }) => (
        <Withdrawal.Funnel
          closeModal={closeModal}
          setCurrentModalMetaInfo={setCurrentModalMetaInfo}
          renderSteps={{
            assetPassword: ({ onNext, closeModal }) => (
              <Withdrawal.AssetPassword onNext={onNext} closeModal={closeModal} />
            ),
            otp: ({ closeModal, withdrawRequestData }) => (
              <Withdrawal.OTP data={withdrawRequestData}
                callbackAfterVerify={verifyAfterAuth} closeModal={closeModal} />
            ),
            email: ({ closeModal, withdrawRequestData }) => (
              <Withdrawal.Email data={withdrawRequestData}
                callbackAfterVerify={verifyAfterAuth} closeModal={closeModal} />
            ),
          }}
        />
      )}
    </VerifyDialog>
  )}
</AssetPasswordRequired>
```

코드가 길어 보이지만, 각 레이어의 역할이 명확하다.

- `AssetPasswordRequired` — 자산 비밀번호 미설정이면 설정 유도 모달을 보여주고, 설정됐으면 children 렌더링
- `VerifyDialog` — 모달 열기/닫기와 title/description을 Step에 따라 동적 변경
- `Withdrawal.Funnel` — Step 전환과 context 전달
- `renderSteps` — 각 Step에서 실제로 뭘 할지는 외부에서 결정

출금 주소 등록도 같은 구조로 `WithdrawalAddress.Funnel`을 만들면 된다. 검증 흐름은 재사용하고, `renderSteps`에 다른 콜백을 넣으면 끝이다.

## AssetPasswordRequired — 가드 컴포넌트

자산 비밀번호 미설정 유저를 걸러내는 컴포넌트도 따로 분리했다. 이걸 Funnel 안에 넣으면 검증 로직과 설정 유도 로직이 섞여서 복잡해진다.

```tsx
export default function AssetPasswordRequired({ open, setOpen, children }: AssetPasswordRequiredProps) {
  const { data: securityConfig } = useQuery({
    queryKey: exchangeQueryKey.user.security(userId).queryKey,
    queryFn: getMySecurityConfig,
    enabled: !!ex && open,
  });

  if (!securityConfig) return null;

  // 미설정이면 설정 유도 모달
  if (!securityConfig.hasAssetPassword) {
    return (
      <RawModal open={open} onOpenChange={setOpen}>
        <p>자산 비밀번호가 설정되어있지 않습니다.</p>
        <Link to={`${currMainPrefix}/exchange/my/security/asset-password`}>확인</Link>
      </RawModal>
    );
  }

  // 설정 완료면 children 렌더링
  return <>{children({ open, setOpen })}</>;
}
```

children을 함수로 받는 이유는, `open`과 `setOpen`을 내려줘야 하기 때문이다. `AssetPasswordRequired`가 보안 설정 상태를 확인한 뒤에야 children에 제어권을 넘긴다.

## 추상화의 기준 — 뭘 고정하고 뭘 열어둘 것인가

이 설계를 하면서 계속 의식한 건, **"이 컴포넌트는 뭘 알아야 하고, 뭘 모르고 있어야 하는가"**였다.

| 컴포넌트 | 아는 것 | 모르는 것 |
|---|---|---|
| `AssetPasswordRequired` | 유저의 보안 설정 상태 | 검증 플로우 내용 |
| `VerifyDialog` | 모달 열기/닫기, title/description | 어떤 Step이 있는지 |
| `Funnel` | Step 순서, Step 간 데이터 전달 | 각 Step의 구체적 렌더링 |
| `OTPInputUI` | OTP 입력 UI 렌더링 | 검증 로직, 에러 처리 |
| `useAssetPasswordVerification` | 에러 코드별 분기, 시도 횟수 | UI, 어떤 Funnel에서 쓰이는지 |

각 레이어가 자기 관심사만 알고 나머지는 모르니까, 새 기능에 검증을 붙일 때 Funnel 하나 추가하고 `renderSteps`에 콜백을 넣으면 된다. 기존 컴포넌트를 수정할 필요가 없다.

## 돌이켜보면

처음부터 이 구조를 설계한 건 아니었다. UI 컴포넌트 → 단일 플로우(자산비밀번호 설정) → 재사용 구조(출금/주소 등록 Funnel) → 확장(코인니스 본체 비밀번호/탈퇴) 순서로 점진적으로 발전시켰다.

처음부터 "재사용 가능한 추상 레이어를 만들자"고 시작했으면 오히려 과도한 설계가 됐을 거다. 구체적인 케이스를 먼저 만들고, 두 번째 케이스가 생겼을 때 공통점을 뽑아내는 게 더 현실적이었다. 추상화의 타이밍은 "비슷한 게 두 번째로 생겼을 때"가 맞다는 걸 다시 확인한 작업이었다.
